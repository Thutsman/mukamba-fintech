# Custom Confirmation Email Runbook

## Objective
Keep a single confirmation email for signup:

- Use the custom confirmation flow in application code (Resend API).
- Prevent Supabase from sending the default "Confirm signup" message.
- Configure Supabase SMTP with Resend credentials for any Supabase-auth emails that still need delivery.

## Current Confirmation Triggers (Code Audit)
Signup currently has two potential senders:

1. **Custom sender (must stay enabled)**  
   - `src/lib/store.ts` calls `POST /api/send-confirmation-email` after `supabase.auth.signUp(...)`.
   - `src/app/api/send-confirmation-email/route.ts` creates token and calls `sendConfirmationEmail(...)`.
   - `src/lib/email-service.ts` sends with `resend.emails.send(...)`.

2. **Supabase default sender (must be disabled for signup confirmation)**  
   - `supabase.auth.signUp(...)` can trigger Supabase's built-in confirm-signup email when enabled in dashboard.

## Required Supabase Dashboard Configuration

### 1) SMTP (use Resend through Supabase)
In **Supabase Dashboard → Authentication → Settings → SMTP Settings**:

- Enable custom SMTP.
- Use Resend SMTP credentials from your Resend account.
- Save and send a test email from Supabase.

### 2) Disable Supabase signup confirmation output
In **Supabase Dashboard → Authentication → Email Templates**:

- Open **Confirm signup** template.
- Disable its delivery for this project flow (project standard: blank/disabled template).

Do not disable the custom API route in the app; the app owns confirmation delivery.

### 3) URL Configuration
In **Supabase Dashboard → Authentication → URL Configuration**:

- Set **Site URL** to your active environment domain.
- Add allowed redirect URLs including:
  - `http://localhost:3000/auth/confirm` (local)
  - `https://<your-domain>/auth/confirm` (production)

## Environment Variables
Required for this flow:

- `RESEND_API_KEY`
- `NEXT_PUBLIC_APP_URL`
- `NEXT_PUBLIC_SITE_URL`
- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`

Optional guard:

- `NEXT_PUBLIC_SUPABASE_CONFIRM_EMAIL_ENABLED=false`  
  Keep this `false` in environments using custom confirmation flow.

## Validation Checklist
Run after dashboard/config updates:

1. Create a new account with an unused email.
2. Confirm exactly one email arrives.
3. Verify sender/content is the Mukamba branded email generated by app flow.
4. Click confirmation link and verify `/auth/confirm` flow completes.
5. Use "Resend confirmation" and confirm only one resend email arrives.
6. Test password-reset flow still sends correctly.

## Rollback
If custom confirmation mail delivery is degraded:

1. Re-enable Supabase "Confirm signup" template temporarily.
2. Keep logging and inspect Resend dashboard delivery logs.
3. Restore single-email mode after issue resolution.
